## Project Overview

This is a Godot 4.6 project (GDScript, C#-capable via .NET) for simulating and visualizing an 8-faced origami robot. The robot model was generated by ACDC4Robot and exported as an SDF (Simulation Description Format) with binary STL mesh files. The primary scene is statically defined in `scenes/origami_robot.tscn` with transforms **strictly computed from `sdf/origami_robot/origami_robot.sdf`** using SDF→Godot coordinate conversion.

## Scene Architecture

### Main Scene (`scenes/origami_robot.tscn`)
- **Root** `OrigamiRobot` (`Node3D`) with `robot_setup.gd` script (loads STL meshes at edit-time/runtime)
- **`MotionController`** (`Node3D`): sibling of all faces, runs `robot_motion.gd` — drives all 8 HingeJoint3D motors with sinusoidal velocity targets for wave-like crawling motion
  - Configurable params: `speed_amplitude` (180°/s), `period` (2s), `phase_offset` (0.125), `motor_max_impulse` (10)
  - Motors are disabled in headless mode (no physics motor support without display server)
- **8× `RigidBody3D`** (`face0`–`face7`): each has `mass = 1.325` and SDF→Godot coordinate-transformed `transform`
  - `visual` (`MeshInstance3D`): mesh populated by `robot_setup.gd` from `.stl`; each face assigned a distinct color
  - `collision` (`CollisionShape3D`): `ConvexPolygonShape3D` via `ArrayMesh.create_convex_shape()` (see [SDF→tscn 导入约束](#sdf-tscn-导入约束))
- **8× `HingeJoint3D`** (`joint0`–`joint7`): connect faces in a closed loop (`face0→face1→…→face7→face0`)
  - Joints are children of their **parent** link, with local transform per SDF 1.7 spec
  - `node_a = ..` (parent), `node_b = ../../child`

### Outer Scene (`scenes/ocean.tscn`)
- `Node3D` root that instances `scenes/origami_robot.tscn` as a child. Sandbox/playground scene.
- **`WaterSurface`** (`MeshInstance3D` + `PlaneMesh` 20×20, semi-transparent blue `StandardMaterial3D`): flat water plane at **y = 0**, NO shader / NO Area3D / NO ripple effect by design.
- **`DirectionalLight3D`** with shadows.
- **`Ground`** (`StaticBody3D`): 20×0.1×20 box at y = -0.05 with `BoxMesh` + `StandardMaterial3D` (dark gray-blue, `albedo_color = Color(0.2, 0.25, 0.3, 1)`, `roughness = 0.8`) and `BoxShape3D` collision. Provides a solid surface for the robot to land on.
- **`Camera3D`**: positioned at y ≈ 1.0, looking down at the robot.
- The robot starts at y ≈ 0.724 and falls onto the water surface / ground.

## tscn 文本格式约束

编辑 `.tscn` 文件时必须遵守 Godot 文本资源的解析顺序：

1. **`[sub_resource]` 必须在 `[node]` 之前定义** — 任何 `[node]` 引用的 `SubResource(id)` 必须在同文件中先出现对应的 `[sub_resource]` 定义；否则加载时报 `!int_resources.has(id)` 错误。
2. **被引用的子资源必须先定义** — 如果 `BoxMesh` 的 `material = SubResource(2)`，则 `[sub_resource type="StandardMaterial3D" id="2"]` 必须出现在 `[sub_resource type="BoxMesh" id="1"]` 之前。
3. **`[ext_resource]` 在最前面** — 外部资源引用紧随文件头之后、任何 `[sub_resource]` 之前。
4. **`[node]` 属性缩进用制表符** — Godot 编辑器保存时使用的格式；子节点路径如 `parent="Ground"` 必须匹配实际父节点名。

## Buoyancy & Water Surface

- **Script**: `scripts/buoyancy/buoyancy_mesh.gd` — ported from [q-qo-o/godot-floating-objects](https://github.com/q-qo-o/godot-floating-objects) (`buoyancy_mesh.gd`). Algorithm: clip each mesh triangle against the water plane `y = water_height`, sum signed tetrahedron volumes `V = (1/6)·r0·(r1×r2)` for submerged parts, apply Archimedes force `F = ρ·g·V_sub` upward at the volume-weighted buoyancy centroid. Requires **closed meshes with outward normals**.
- **Attached to all 8 faces** in `scenes/origami_robot.tscn` (each face RigidBody3D gets `script = buoyancy_mesh.gd`).
- **Water height source**: the script reads `water_surface_node.global_position.y`. If `water_surface_node` is null it auto-finds a node named **`WaterSurface`** in the current scene — so the outer scene's `WaterSurface` node MUST keep that exact name. The lookup is null-guarded (won't crash if `get_tree().current_scene` is null).
- **`buoyancy_multiplier`** (default `1.0` = pure Archimedes; set to `2.0` on every face): the origami faces are thin shells (STL = closed mesh, 8 tris / 6 verts, solid volume ≈ 1.25e-3 m³, mass 1.325 kg → density ≈ 1060 kg/m³, slightly negative buoyancy in water ρ=1000). The real displaced volume (outer envelope) exceeds the solid mesh volume, so a multiplier > 1 gives physically reasonable positive buoyancy. Tune per-face in the inspector.
- **Why mesh-based, not voxel-based**: the voxel script (`buoyancy.gd` in the reference repo) only supports `BoxShape3D`; these faces use `ConvexPolygonShape3D`, so the mesh-geometry version is required. Verified each STL is a closed manifold (0 non-manifold edges) so the signed-volume method is valid.
- **Headless note**: `_integrate_forces` runs in headless mode fine, but the buoyancy only matters visually with a real water surface.

### SDF Import (Backup / Addon)
- `sdf/origami_robot/origami_robot.tscn`: auto-generated by the editor addon (`addons/sdf_importer/`), separate from the hand-authored main scene
- The `sdf_loader.gd` script (not used by any scene) dynamically builds nodes from SDF XML at runtime
- Both use the same SDF→Godot coordinate math (COORD_T basis, STL parsing, axis alignment)

## SDF Importer Addon (`addons/sdf_importer/`)
- **Editor plugin**: adds "Import SDF as tscn" under the Project → Tools menu; opens a file picker for `.sdf` files
- **SDFParser**: regex-based XML parser extracting model name, links (pose, mass, mesh URI/scale, inertial pose), and joints (type, parent/child, pose, axis)
- **STLParser**: standalone `RefCounted` class (`class_name STLParser`) for binary STL → `ArrayMesh` with SDF→Godot coordinate conversion
- **SDFImporter**: `RefCounted` class (`class_name SDFImporter`); orchestrates parsing → scene tree construction → `PackedScene` pack → `ResourceSaver.save()` to `.tscn`
- Both parser classes use `class_name` so they can be `new()`'d from any script

## Coordinate Systems

- **SDF**: right-handed, Z-up
- **Godot**: left-handed, Y-up
- **Position/Vertex conversion**: `(x, y, z)_sdf` → `(x, z, -y)_godot`
- **Basis conversion**: `B_godot = C * B_sdf * C^T` where `C = [X→X, Y→-Z, Z→Y]` (expressed in Godot basis as `Basis(Vector3(1,0,0), Vector3(0,0,-1), Vector3(0,1,0))`)
- **Mesh scale**: SDF specifies `<scale>0.001 0.001 0.001</scale>`; applied in all scripts via `mesh_scale = 0.001`

## SDF→tscn 导入约束

将 SDF 描述文件导入为 `.tscn` 时，必须遵守以下硬性约束（适用于 `robot_setup.gd`、`sdf_loader.gd`、`addons/sdf_importer/sdf_importer.gd` 三处导入代码）：

1. **碰撞形状用 `ConvexPolygonShape3D`，不用 `ConcavePolygonShape3D`** — Godot 规定 `ConcavePolygonShape3D` 应配合 `StaticBody3D` 等静态 `CollisionObject3D` 使用；与 `RigidBody3D` 一起无法正常工作（除非冻结或 `freeze_mode = FREEZE_MODE_STATIC`）。这些面是 `RigidBody3D`，必须用 `ArrayMesh.create_convex_shape()`。
2. **材质 `cull_mode` 必须设为 `CULL_DISABLED`** — 折纸面是薄壳（STL = 闭合网格但两面都需可见），创建 `StandardMaterial3D` 后必须设 `mat.cull_mode = BaseMaterial3D.CULL_DISABLED`，否则背面会被剔除看不见。
3. **依据 SDF 中的 `<uri>` 加载对应 STL** — SDF `<visual>`/`<collision>` 的 `<uri>` 形如 `model://origami_robot/meshes/faceN.stl`，需解析路径（`model://` 前缀替换为 `res://sdf/`，或用 `sdf_dir.path_join(...)`）后用 `STLParser.parse_stl()` 置入场景的 visual/collision 节点。导入器必须真正读取并实例化这些 mesh，而非空节点。
4. **`COORD_T` 必须是列存储的正确顺序** — Godot `Basis` 是**列存储**：`Basis * v = v.x·col0 + v.y·col1 + v.z·col2`。正确写法是 `Basis(Vector3(1,0,0), Vector3(0,0,-1), Vector3(0,1,0))`，使 `(x,y,z)_sdf → (x,z,-y)_godot`。若把 col1/col2 写反（如旧的 `Basis((1,0,0),(0,0,1),(0,-1,0))`），位置 Y 分量符号会反、旋转也会错。
5. **无双重网格缩放** — `parse_stl()` 已将 `mesh_scale=0.001` 应用于顶点。visual/collision 节点**不可**再设 `scale = Vector3(...)` 变换，否则双重缩放使网格变小 1,000,000×。
6. **`HingeJoint3D` 电机用嵌套属性路径（Godot 4.6）** — `motor/enable`、`motor/target_velocity`、`motor/max_impulse` 必须用 `joint.set("motor/target_velocity", val)` 设置；扁平属性 `motor_target_velocity` 等已不存在。
7. **`parse_stl()` 参数名勿用 `scale`** — 会遮蔽继承自 `Node3D` 的 `scale` 属性触发警告，参数命名为 `mesh_scale`。

## Video Export (Movie Maker)

Godot 4.6 has built-in Movie Maker (`--write-movie`) — no plugin needed. Export a 20s video at default viewport resolution (1152×648) / 60 FPS:

```bash
# 1) Render to AVI (MJPEG). 1200 frames = 20s @ 60 FPS. DO NOT pass --headless.
"$GODOT" --path . --write-movie output/videos/ocean_temp.avi --quit-after 1200

# 2) Transcode to MP4 with timestamp in filename
TIMESTAMP=$(date +%Y-%m-%d_%H-%M-%S)
ffmpeg -y -i output/videos/ocean_temp.avi -c:v libx264 -preset medium -crf 18 \
  -pix_fmt yuv420p -movflags +faststart "output/videos/ocean_${TIMESTAMP}.mp4"

# 3) Clean up temp AVI
rm output/videos/ocean_temp.avi
```

### Output conventions
- **Output directory**: `./output/videos/` (create with `mkdir -p` before running).
- **Filename format**: `ocean_YYYY-MM-DD_HH-MM-SS.mp4` — always include the current timestamp so successive exports don't overwrite each other.
- **Resolution**: Movie Maker captures the root viewport size (project default is 1152×648). `--resolution` is ignored in movie mode; set via `display/window/size/viewport_width|height` in `project.godot` or a temporary `override.cfg`.
- **Frame rate**: Default 60 FPS. Use `--fixed-fps N` to override.

### Hard constraints (gotchas)
- **`--quit-after` counts FRAMES, not seconds.** Pass `1200` for 20s @ 60 FPS, not `20`. Passing `20` produces a 0.33s video; passing `20000` produces a 5.5 min video.
- **`--write-movie` CANNOT be combined with `--headless`.** Headless forces the dummy rendering driver, which cannot readback GPU textures → crash (`Parameter "t" is null` in `texture_2d_get`). Always run Movie Maker with the real display driver (d3d12 on Windows).
- **Real display driver ⇒ motors are ENABLED.** Because `DisplayServer.get_name() != "headless"`, `robot_motion.gd`'s motor guard passes and the robot does its sinusoidal wave drive while floating. Expect motion in the video (this is intended, not a bug).
- **Godot `--write-movie` only supports `.avi` and `.png`** — it cannot write MP4 directly. Always transcode with ffmpeg after recording.
- **`--write-movie` output path is relative to the project directory** (the `--path` argument), not the current working directory.
- **No editor import step needed**: the writer works from a plain run (`--path . --write-movie ...`), not `--editor`.

## Key Technical Details

- **Physics**: Jolt Physics (`3d/physics_engine="Jolt Physics"`)
- **Rendering**: Forward Plus; Windows driver = d3d12
- **STL parsing**: Uses `StreamPeerBuffer` with `big_endian = false` to read binary STL little-endian floats; converts vertices/normals from SDF to Godot coordinate space before building an `ArrayMesh`
- **@tool mode**: `robot_setup.gd` runs in the editor so meshes appear immediately when opening the scene
- **Joint transforms**: In the main static scene, joints are children of their **parent** link with pre-computed local transforms. In the dynamic importers (`sdf_loader.gd`, `sdf_importer.gd`), joints are children of **root** and use world-space transforms. Joint `<pose>` in SDF 1.7 is relative to the **child** link frame (per the dynamic importers). Joint `<axis>` with `expressed_in="__model__"` is in model root frame, transformed to Godot world via COORD_T.
- **Motion controller**: Drives HingeJoint3D motors with sinusoidal velocity profile `v_i(t) = amplitude * sin(2πt/T + i * phase_offset * 2π)`. Motors are disabled in headless mode (`DisplayServer.get_name() == "headless"`). **Consequence**: any non-headless run (including Movie Maker video export) enables the motors — see [Video Export (Movie Maker)](#video-export-movie-maker).
- **No C# code yet**: Project has .NET/C# config enabled (`project.godot` has `[dotnet]` section) but no `.cs` files exist — all scripts are GDScript
